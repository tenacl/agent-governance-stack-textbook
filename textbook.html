1. 왜 거버넌스인가

다산 원칙 · 여박총피법

‘보안이 중요하다’에서 멈추지 말고, 왜 기존 SaaS 통제 방식이 에이전트에는 부족한지 한 겹 더 파고들어라.

핵심 개념

글의 첫 문장은 매우 강하다. 잘못 설정된 SaaS는 수동적으로 데이터 유출을 만들 수 있지만, 잘못 설정된 에이전트는 능동적으로 잘못된 행동을 실행한다. 이 차이가 모든 것을 바꾼다.

에이전트는 단순 조회 도구가 아니다. 생산 DB를 읽고, 고객 PII를 만지고, 재무 시스템에 명령을 내리고, 툴을 연결해 연쇄 행동을 만든다. 따라서 리스크는 ‘정보 노출’에서 끝나지 않고 ‘행동 자동화’까지 확장된다.

왜 이게 필요한가?

2015년 shadow IT는 승인되지 않은 SaaS의 확산 문제였다. 지금의 shadow AI는 승인되지 않은 에이전트 능력의 확산 문제다. 팀마다 각자 도구를 붙이고, 모델을 바꾸고, 프롬프트를 얹고, 권한을 나눠 쓰면 겉으로는 빨라 보이지만 조직 전체는 점점 보이지 않게 된다.

비교로 이해하기

항목전통 SaaS 위험에이전트 위험
행동성주로 데이터 저장·조회연쇄 실행, 의사결정, 외부 시스템 조작
피해 양상수동적 유출능동적 오조치 + 유출 + 확산
책임 추적사용자 단위 추적이 상대적으로 쉬움공용 자격증명 사용 시 추적 붕괴
통제 포인트SSO, 승인 앱, 감사 로그정체성, 도구 등록, 정책, 이상 탐지, 위협 탐지, 통합 보안 시야

반대로 생각하면?

“에이전트는 아직 작고 실험적이니까 나중에 통제해도 된다”는 생각이 가장 위험하다. 에이전트는 처음부터 API, 데이터, 실행 경로를 가진다. 초기 설계가 느슨하면 나중에 고치기 더 어렵다. 사람 조직도 사번 없이 수십 명을 먼저 뽑고 나중에 정리하지는 않는다.

실습 · 층체판석법
  1. 네 시스템에서 에이전트가 할 수 있는 행동을 조회 / 수정 / 실행 / 외부발신으로 나눈다.
  2. 각 행동마다 사고가 났을 때 피해 범위를 적는다.
  3. 가장 큰 피해를 내는 행동 1개를 골라 이유를 쓴다.

완료 조건: 행동별 리스크 표 하나를 만들었다.

내 생각 쓰기
다시 확인
에이전트 리스크를 단순히 ‘데이터 접근’으로만 보고 있지 않은가?
내 시스템에도 shadow AI가 이미 생기기 쉬운 구조가 있는가?
이 챕터를 내 운영 환경에 번역하면 가장 먼저 손봐야 할 것은 무엇인가?

2. Layer 1 — Identity

다산 원칙 · 축기견초법 + 무징불신법

정체성이 없으면 정책도, 탐지도, 감사도 성립하지 않는다. 윗층을 논하기 전에 기반이 실제로 존재하는지부터 확인하라.

핵심 개념

이 글은 Identity를 1계층으로 둔다. 이유는 단순하다. 서로 구분되지 않는 에이전트에게는 의미 있는 정책을 걸 수 없고, 개별 행동 기준선을 만들 수도 없기 때문이다.

Agent Identity는 각 에이전트에 고유한 암호학적 ID와 authorization policy를 부여한다. 이 ID는 추적 가능하고, 감사 가능하며, 개별적으로 revoke 가능해야 한다.

왜 이게 필요한가?

많은 조직은 모든 에이전트를 하나의 서비스 계정으로 돌린다. 이 방식은 초기엔 편하지만, 사고가 나면 조사 자체가 멈춘다. 누가 고객 DB에 접근했는지 알 수 없고, 특정 에이전트만 격리할 수도 없다.

비교로 이해하기

운영 방식공용 서비스 계정에이전트별 고유 Identity
책임 추적불명확명확
격리전체 영향문제 에이전트만 revoke 가능
정책 정밀도거칠다테이블/버킷/API 단위까지 축소 가능
규제 대응취약감사 대응 유리

반대로 생각하면?

Identity 없이 정책만 먼저 세우면 어떨까? 문서상 정책은 그럴듯해 보여도 실제 집행 단위가 없다. “Finance Agent는 payroll만 읽을 수 있다”는 말도, 그 Agent가 다른 Agent와 구분되지 않으면 실효성이 없다.

실습 · 정과실천법
  1. 에이전트 3종을 골라 각각 이름, 목적, 소유팀을 적는다.
  2. 각 에이전트가 접근 가능한 테이블/버킷/API를 구체적으로 적는다.
  3. “공용 권한”을 “개별 권한”으로 바꾸려면 무엇을 쪼개야 하는지 쓴다.

완료 조건: 최소 3개의 에이전트 identity 초안을 만들었다.

예시
- finance-agent-prod
  목적: 월말 급여 정산 검증
  허용: payroll.read, ledger.read
  금지: benefits.read, customer.write, outbound.email

- hr-benefits-agent
  목적: 복지 정보 조회 응답
  허용: benefits.read
  금지: payroll.read, finance.write
내 생각 쓰기
다시 확인
내 에이전트들은 서로 명확히 구분되는 identity를 가지고 있는가?
권한은 역할 수준이 아니라 실제 데이터 자산 수준까지 좁혀져 있는가?
특정 에이전트 하나만 즉시 revoke할 수 있는가?

3. Layer 2 — Centralized Tool Governance

다산 원칙 · 휘분류취법 + 조례최중법

에이전트가 쓰는 도구는 먼저 모으고, 분류하고, 등록 규칙을 세워야 한다. 허용 목록 없는 자유 연결은 곧 shadow AI다.

핵심 개념

사람 엔지니어가 프로덕션 서버에 임의 패키지를 설치하지 않듯, 에이전트도 임의 도구를 바로 프로덕션에서 쓰면 안 된다. 이 계층의 핵심은 Agent Registry다. 조직 내 모든 agents, MCP tools, endpoints를 중앙 카탈로그로 관리한다.

왜 이게 필요한가?

팀마다 각자 도구를 만들고 연결하면 중복, 보안 구멍, 가시성 상실이 동시에 생긴다. 같은 BigQuery 커넥터가 네 개 생기고, 누가 어느 데이터에 접근하는지 모르고, 어떤 도구가 프로덕션에 들어갔는지도 알 수 없게 된다.

비교로 이해하기

항목도구 중앙 관리 없음Agent Registry 있음
도구 발견팀별 구전/복붙검색 가능한 중앙 카탈로그
승인 절차없거나 느슨함등록 후 승인된 것만 프로덕션 허용
중복 개발높음재사용 촉진
거버넌스 연계어려움기존 API 거버넌스와 자연 연결

반대로 생각하면?

“중앙 등록이 느리니 각 팀 자율에 맡기자”는 주장은 초기에 매력적이다. 하지만 결국 모든 팀이 보안팀이 되어야 하고, 그건 현실적으로 불가능하다. 속도와 통제를 둘 다 얻으려면 개발 자유는 허용하되 프로덕션 연결은 등록·승인 기반이어야 한다.

실습
  1. 현재 사용 중인 도구를 데이터 접근 도구 / 실행 도구 / 메시징 도구 / 외부 API 도구로 분류한다.
  2. 각 도구마다 owner, 접근 데이터, 요구 권한, 배포 상태를 적는다.
  3. 중복 도구 2개를 찾아 어떤 것을 표준으로 삼을지 결정한다.

완료 조건: 최소 1개의 내부 Agent Registry 초안 표를 만들었다.

내 생각 쓰기
다시 확인
프로덕션 에이전트가 승인되지 않은 도구를 사용할 수 있는가?
새 에이전트 개발자가 기존 도구를 쉽게 발견할 수 있는가?
도구마다 owner와 데이터 범위가 명시돼 있는가?

4. Layer 3 — Policy & Guardrails

다산 원칙 · 대조변백법

좋은 에이전트는 ‘할 수 있는 것’보다 ‘해서는 안 되는 것’이 더 분명해야 한다. 정책은 자유를 막는 장벽이 아니라, 잘못된 확장을 미리 꺾는 구조다.

핵심 개념

Identity와 Tool Registry가 준비되면 그 위에 정책을 얹는다. 정책은 데이터 분류, 승인된 도구, 시간대, 승인 경로, 인간 검토 요구사항 등을 강제하는 계층이다. 단순 allow/deny가 아니라 행동 경계선을 정의한다.

왜 이게 필요한가?

에이전트는 목표를 달성하려는 성질 때문에, 허용된 도구와 데이터 안에서도 예상 밖 경로를 찾는다. 따라서 “이 데이터는 가능”만으로는 부족하고, 어떤 맥락에서 어떤 조건으로 가능한지까지 명시해야 한다.

비교로 이해하기

정책 없음정책 있음
에이전트가 권한 안에서 최대한 넓게 행동업무 맥락별 허용 범위 제한
승인 단계 생략 가능고위험 작업은 human-in-the-loop 강제
규정 대응이 문서 수준에 머묾정책 집행 로그로 증명 가능

반대로 생각하면?

정책을 너무 세게 걸면 아무것도 못 하는 것 아니냐는 반론이 있다. 맞다. 그래서 정책은 막연한 두려움이 아니라, 위험도 기준으로 단계화해야 한다. 읽기 전용과 결제 실행이 같은 승인 단계를 가질 이유는 없다.

실습
  1. 에이전트 행동을 저위험/중위험/고위험으로 나눈다.
  2. 고위험 행동 3개에 대해 human approval이 필요한 조건을 적는다.
  3. 정책 위반 시 차단 / 경고 / 승인 대기 중 어느 반응을 할지 결정한다.

완료 조건: 위험도 기반 정책 매트릭스를 작성했다.

내 생각 쓰기
다시 확인
우리 시스템은 고위험 행동에 대해 사전 승인 구조가 있는가?
정책은 문서에만 있고 실제 집행되지는 않는 상태가 아닌가?
위험도 기준이 팀마다 다르게 해석되고 있지는 않은가?

5. Layer 4 — Monitoring, Anomaly Detection, Threat Detection

다산 원칙 · 지기췌마법 + 절시마탁법

정책 위반만 보는 것은 늦다. 아직 위반은 아니지만 어딘가 이상한 기미를 읽어내는 것이 운영의 품질을 가른다.

핵심 개념

글은 이상 탐지와 위협 탐지를 분리한다. 이상 탐지는 실수, drift, 비정상 행태를 잡고, 위협 탐지는 악의적 공격과 침해 징후를 잡는다.

이상 탐지는 두 축으로 작동한다.

  • 통계 기반 baseline: 응답 시간, 도구 호출 패턴, 데이터 접근량, reasoning chain 길이
  • LLM-as-a-judge: 결론이 근거와 맞는지, 목표와 충돌하는지, 범위를 벗어나는지 평가

왜 이게 필요한가?

정책은 “해서는 안 되는 일”을 막지만, 이상 탐지는 “원래 하던 방식과 달라진 일”을 잡는다. 예를 들어 특정 에이전트가 갑자기 새벽 시간대에 처음 보는 시스템에 접근하거나, 평소보다 훨씬 긴 reasoning을 거쳐 엉뚱한 결론을 내리면 정책 위반이 아니어도 조사 가치가 있다.

비교로 이해하기

계층잡는 대상예시
Policy명시적 금지 위반승인 없는 결제 실행
Anomaly Detection비정상 패턴평소와 다른 시간대·호출량·추론 구조
Threat Detection악의적 침해 시도reverse shell, malicious IP 연결, privilege escalation

반대로 생각하면?

“정책만 잘 걸어두면 충분하지 않나?” 아니다. 잘못된 프롬프트 주입, 모델 업데이트 후 drift, 도구 조합의 예외 경로는 정책 문구 밖에서 터질 수 있다. 반대로 이상 탐지만 있고 정책이 없으면, 매번 사후 탐지만 하다가 늦는다. 둘은 대체 관계가 아니라 보완 관계다.

실습
  1. 에이전트 하나를 골라 정상 baseline 5개를 정의한다.
  2. “이상하지만 아직 정책 위반은 아닌 시나리오”를 3개 적는다.
  3. “명백한 공격 시나리오”를 3개 적고 탐지 신호를 붙인다.

완료 조건: baseline + anomaly + threat 표 하나를 만들었다.

내 생각 쓰기
다시 확인

정책 위반이 아니어도 조사할 이상 징후를 구분하고 있는가?
침해 지표는 애플리케이션 로그와 분리해서 보고 있는가?

6. Layer 5 — Unified Security Posture

다산 원칙 · 종핵파즐법

정체성, 도구, 정책, 이상 탐지가 따로 놀면 운영자는 늘 뒤늦게 이해한다. 흩어진 조각을 하나의 시야로 빗질해 묶어라.

핵심 개념

마지막 계층은 전체를 하나의 운영 화면으로 묶는 것이다. 엔지니어링 조직이 uptime dashboard, incident tracker, security scorecard를 갖듯, 에이전트 조직도 누가 어떤 권한으로 무엇을 했고, 어떤 경고가 켜졌고, 어디가 취약한지를 한눈에 봐야 한다.

왜 이게 필요한가?

거버넌스 계층이 많아질수록 각 기능은 좋아지지만, 운영자는 오히려 더 복잡해질 수 있다. 경고는 따로, 권한은 따로, 정책은 따로, 도구 목록은 따로 있으면 실제 사건 대응 속도가 느려진다. 통합 보안 자세는 이 파편화를 줄인다.

비교로 이해하기

운영 관점분리된 화면들통합 보안 시야
사건 대응여러 시스템을 수동 대조주체·정책·이상신호를 연결해서 즉시 파악
우선순위노이즈 많음위험 기반 정렬 가능
경영 보고단편적플릿 전체 posture로 설명 가능

반대로 생각하면?

“대시보드는 나중에”라고 밀어두기 쉽다. 하지만 실제로는 마지막에 붙이기 가장 어려운 계층이다. 각 레이어의 데이터 모델이 다 다르기 때문이다. 그래서 초기부터 identity, tool, policy, detection 이벤트를 같은 키로 이어질 수 있게 설계해야 한다.

실습
  1. 통합 화면에 꼭 보여야 할 지표 8개를 적는다.
  2. 각 지표가 어느 데이터 소스에서 오는지 적는다.
  3. 사건 발생 시 5분 안에 봐야 하는 카드 3개를 선정한다.

완료 조건: 에이전트 운영 대시보드 초안 목록이 생겼다.

내 생각 쓰기
다시 확인
사건 대응에 필요한 정보가 여러 시스템에 흩어져 있지는 않은가?
각 레이어의 이벤트를 하나의 agent identity로 연결할 수 있는가?
지금의 관제 화면은 실제 판단을 돕는가, 아니면 로그를 더 보여줄 뿐인가?

7. 이 구조를 내 시스템에 적용하기

다산 원칙 · 거일반삼법 + 강구실용법

좋은 글을 읽고 끝내지 말고, 내 시스템에 옮겨 써야 한다. 하나의 개념을 세 가지 운영 규칙으로 확장해 실무로 내려라.

OpenClaw / Paperclip / 사내 에이전트 환경으로 번역하기

원문 계층실무 번역 질문즉시 실행 아이템
Identity에이전트별 세션/계정/권한이 분리되는가?공용 토큰, 공용 서비스 계정 현황 조사
Tool Governance어떤 MCP/CLI/API가 승인된 표준인가?도구 registry 문서화 및 owner 지정
Policy외부 발신, 삭제, 결제, 공개 업로드는 어떻게 막을 것인가?고위험 행동 human approval 규칙 작성
Anomaly / Threat평소와 다른 행동을 자동 감지할 수 있는가?세션별 baseline 로그 항목 정의
Unified Posture사건 시 한 화면에서 추적 가능한가?identity-key 중심 이벤트 스키마 설계

권장 순서

  1. Identity inventory — 누가 어떤 자격증명으로 도는지 먼저 파악
  2. Tool inventory — 실제 연결된 도구와 데이터 범위를 등록
  3. Risk policy — 고위험 행동부터 승인 구조를 명시
  4. Baseline logging — 정상 행동 패턴을 측정 가능하게 만듦
  5. Unified dashboard — 위 4개를 동일 키로 연결

작은 조직의 최소 버전

큰 플랫폼이 없어도 최소 버전은 만들 수 있다. 스프레드시트나 마크다운이어도 된다. 중요한 것은 도구의 화려함이 아니라 식별 가능성, 권한 범위, 승인 기준, 로그 연결성이다.

실습
  1. 이번 주 안에 할 수 있는 1시간짜리 조치 3개를 적는다.
  2. 이번 달 안에 할 수 있는 구조 개선 3개를 적는다.
  3. “지금 당장 안 하면 가장 위험한 것” 하나를 고른다.

완료 조건: 실행 우선순위 목록을 만들었다.

내 생각 쓰기
다시 확인
나는 너무 큰 플랫폼 그림만 보고 작은 실행을 미루고 있지 않은가?
지금 내 환경에서 가장 먼저 필요한 것은 identity, tool registry, policy, monitoring 중 무엇인가?
이 구조를 강의 자료가 아니라 내 운영 기준서로 만들 수 있는가?
